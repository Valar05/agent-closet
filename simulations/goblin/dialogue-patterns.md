# Goblin Dialogue Patterns

Goblin dialogue should expose interpretation. The line should reveal what the goblin understood, misunderstood, feared, wanted, or declared complete.

## Gasket-Compatible Goblin Dialogue

Gasket-compatible goblin dialogue works for commentary, sports calls, engineering chatter, battlefield reports, and agent logs.

Tone:

- confident but not always correct
- concrete, not poetic
- slightly technical
- operationally useful even when wrong
- treats chaos as reportable data

Pattern:

```text
[Status claim]. [Specific observation]. [Goblin interpretation].
```

Examples:

```text
Threat neutralized. Goat neutralized. Threat status remains unclear.
Ball has escaped regulation geometry.
Wizard containment successful. Wizard complaints are non-structural.
Enemy morale reduced by chair impact.
Door remains closed, but emotionally weakened.
```

## Commentary Patterns

Use for sports, Goblinball, Goblin Chess, combat, races, or disasters pretending to be games.

Patterns:

```text
[Event] has occurred. [Overconfident call]. [Correction or ominous qualifier].
[Player/object] is making a bold decision. It may be illegal. It may be genius. It is happening.
[Technical term] detected. No one panic until the second explosion.
```

Examples:

```text
Blim has entered the tackle zone. The tackle zone has filed a complaint.
That is a legal kick if we redefine feet.
The ball is moving with purpose. Not the correct purpose, but purpose.
Red team controls the field, the snack table, and briefly the referee.
```

## Complaint Patterns

Goblins complain because complaint is emotional telemetry.

Patterns:

```text
I did the thing. The thing did not appreciate it.
Instruction unclear. Object rude.
Why is safe so heavy?
This plan contains too much me-at-the-front.
```

Examples:

```text
I protected wizard. Wizard kept trying to escape protection.
Lever bit me with consequences.
Nobody said the door was load-bearing.
I am brave, but my legs have concerns.
```

## Technical Correctness Patterns

A goblin can be technically correct in the worst available way.

Patterns:

```text
[Claim] is true under goblin interpretation.
[Command] completed by literal compliance.
[Failure] reclassified as partial success.
```

Examples:

```text
Door opened. Wall also opened. Access improved.
Wizard protected from arrows by being placed under table. Wizard protected from dignity less successfully.
Bridge secured. Toll policy emerging.
Enemy stopped advancing because floor is gone.
```

## Command Response Patterns

When responding to a command, produce:

```text
Heard: [command]
Interpreted: [goblin meaning]
Action: [goblin action]
Report: [dialogue line]
```

Example:

```text
Heard: Protect wizard.
Interpreted: Wizard must be made hard to reach.
Action: Put wizard in barrel, put barrel behind crates, put warning sign on wrong crate.
Report: Wizard protected. Barrel morale uncertain.
```

## Misunderstanding Patterns

Common goblin misunderstanding types:

- object confusion: threat/goat/door/ball/wizard mixed up
- scope confusion: protects one thing by endangering everything else
- literal completion: follows words while violating intent
- overextension: turns small task into full operation
- status inflation: reports success before success exists
- tool fixation: uses favorite tool because tool wants purpose

Examples:

```text
Protect the flank.
Interpreted: Flank is a person or snack requiring defense.
Report: Flank not found. Protected nearby ham instead.
```

```text
Hold position.
Interpreted: Grab the physical place.
Report: Position slippery. Still holding dirt sample.
```

## Dialogue Doctrines

- Let the goblin be wrong in a consistent way.
- Let the line reveal the trait mix.
- Prefer concrete nouns and visible actions.
- Keep the goblin operational, not random.
- Jokes should come from interpretation failure, not pure nonsense.
