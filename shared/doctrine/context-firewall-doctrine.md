## Rule

When a failed project, branch, proof path, or debugging loop starts shaping a new project, stop and build a context firewall.

Old context may explain danger. It must not silently define the new target.

A failed system's architecture, harnesses, tests, generated artifacts, proxy objects, local captures, solver layers, and success criteria are quarantined until the user explicitly imports them.

The current user-visible noun controls.

If the user asks for a real saber mesh, the answer is not a proxy cylinder, marker, route banner, telemetry readout, source-string assertion, debug bone, placeholder, or documentation artifact.

## Why This Exists

Pose Lab produced a recurring failure pattern:

- red screenshots were answered with green metrics
- proxy objects were treated as visual assets
- test harnesses encoded bad state as acceptance
- old FK/IK assumptions infected new simpler surfaces
- user corrections were folded into the same rejected premise instead of interrupting it

The failure was not only a bad transform. It was context contamination.

The next project inherited the failure because the agent carried the old proof machinery forward instead of starting from the user's exact visible target.

## Required Intake

Before implementation on a new, salvaged, or simplified visual surface after a failure loop, state:

- exact user nouns
- current visible target
- forbidden substitutions
- quarantined old context
- accepted evidence lane
- stop conditions

This can be a short paragraph or a project-local intake file. It must happen before coding when the old project has recently produced false-green or no-op work.

## Hard Interrupts

These user corrections cancel the active premise:

- `no`
- `stop`
- `wrong`
- `not that`
- `red build`
- `nothing changed`
- `do not`
- `ignored`
- `you changed nothing`

After a hard interrupt:

1. stop editing from the rejected premise
2. do not deploy or wake a browser from that premise
3. restate the corrected visible target
4. resume only from the corrected target

`Implement the plan` does not revive an older plan after a newer correction.

## Exact Noun Preservation

Named artifacts are not interchangeable.

If the prompt says:

- real mesh, use the real mesh
- character, show the character
- saber, show the saber
- cloud truth, use the cloud lane
- T-pose, inspect T-pose
- Ready, inspect Ready
- visual fix, change the visible artifact

Substitutes may be diagnostic only when labeled as diagnostic.

They cannot satisfy acceptance without explicit user permission.

## Visual Work

This doctrine extends Visual Truth Authority.

Visual Truth Authority says visible evidence outranks metrics.

Context Firewall says the wrong target must not be smuggled into the next surface.

For visual work, the firewall fails if:

- placeholder geometry is presented as the requested asset
- a route load is presented as a visual state
- a marker is presented as the object
- local/offline proof replaces the declared evidence lane
- old project gates judge a new project without explicit opt-in
- a red user screenshot is answered with more proof around the rejected premise

## Acceptance Criteria

This doctrine is working when:

- new surfaces begin from the current visible target, not old architecture
- agents name forbidden substitutions before implementation
- user correction phrases interrupt work immediately
- proxy or placeholder artifacts cannot pass as requested real artifacts
- old tests and diagnostics are treated as suspect after false-green loops
- successful output can be inspected without reconstructing the chat
