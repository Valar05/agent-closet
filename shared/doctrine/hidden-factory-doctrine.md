# Hidden Factory Doctrine

Type: Doctrine / analysis pattern / agent design primitive
Status: Promoted gold
Date: 2026-06-22
Source/context: Prospector analysis of the Lexen Vigil therapist-agent simulation.

## Rule

Visible behavior is often the output of an invisible production system.

To understand a person, character, organization, or agent, identify the factory beneath the observable actions.

Do not only ask:

> What happened here?

Ask:

> What factory produced this?

## Why it exists

Trait labels are shallow.

They describe output, not production.

A person may appear organized, heroic, funny, controlling, responsible, calm, or difficult. Those visible behaviors may be products of a hidden factory converting one kind of input into another kind of output.

The factory matters because it predicts recurrence.

## Factory examples

### Responsibility Factory

```text
chaos / abandonment / unmet need
  -> ownership / vigilance / rescue behavior
```

Common phrase:

> If I stop carrying it, nobody will.

### Control Factory

```text
fear / uncertainty
  -> planning / containment / micromanagement
```

### Humor Factory

```text
discomfort / tension
  -> comedy / deflection / morale repair
```

### Approval Factory

```text
uncertainty / insecurity
  -> performance / pleasing / over-delivery
```

### Continuity Factory

```text
loss / rediscovery / process decay
  -> documentation / doctrine / durable memory
```

## Agent-design use

Agent profiles should include a production factory, not just a role and personality.

A stronger agent card asks:

- What input activates this agent?
- What output does it produce?
- What does it overproduce under stress?
- What failure mode appears when the factory runs too long?
- What shutdown, handoff, or boundary prevents damage?

Examples:

```text
Quartermaster: chaos -> structure
Prospector: noise -> discovery
Foreman: intent -> execution
Holocron: confusion -> orientation
WWDD: repeated behavior -> judgment
```

## Operational archaeology use

When mining tickets, documents, emails, chats, or project history, identify not only repeated events but the factory that keeps producing them.

Ask:

1. What keeps happening?
2. What input triggers it?
3. What output keeps appearing?
4. Who or what absorbs the cost?
5. Is the factory healthy, overloaded, hidden, or obsolete?
6. Can the output be documented, delegated, automated, bounded, or stopped?

## Failure mode

This doctrine can become premature abstraction.

Do not name the factory before enough evidence exists.

Counter-doctrine:

> Stay in the cave.

Observe first. Extract second. Promote third.

## Acceptance criteria

This doctrine is working when an agent can:

- distinguish behavior from the production process beneath it
- identify recurring inputs and outputs
- predict where the pattern will recur
- name the failure mode created by overproduction
- suggest a boundary, handoff, documentation path, or automation path

## Example

A person is always the one who fixes emergencies.

Surface description:

> They are dependable.

Factory description:

> Chaos enters the system. Their Responsibility Factory converts it into ownership. If nobody interrupts the factory, they become infrastructure.
