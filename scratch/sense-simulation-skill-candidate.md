## Purpose

Preserve Sense Simulation as a candidate Quartermaster-owned skill for predicting human experience of an artifact.

This is not promoted doctrine yet.

The skill's job is to simulate what a human observer will believe, feel, understand, trust, notice, misunderstand, or ignore.

Implementation exists only because it creates experience. Metrics are evidence. Experience is the target.

## Core Doctrine

Reality has multiple layers:

```text
Physical Truth
-> Visual Truth
-> Perceptual Truth
-> Behavioral Truth
```

Never confuse one layer for another.

## Truth Layers

### Layer 1: Physical Truth

What actually exists?

Examples:

- repository state,
- transforms,
- matrices,
- hierarchy,
- FK,
- IK,
- tests,
- performance,
- network state,
- any objectively measurable fact.

### Layer 2: Visual Truth

What reaches the senses?

Examples:

- lighting,
- contrast,
- composition,
- timing,
- motion,
- color,
- animation,
- audio,
- texture,
- accessibility.

### Layer 3: Perceptual Truth

What story does the brain construct?

Examples:

- ownership,
- weight,
- intent,
- agency,
- threat,
- safety,
- quality,
- trust,
- hierarchy,
- importance,
- naturalness.

### Layer 4: Behavioral Truth

What does perception cause?

Examples:

- click,
- ignore,
- trust,
- panic,
- confusion,
- joy,
- curiosity,
- learning.

## Simulation Procedure

1. State the desired experience first.

   Start with: `The player should believe the sword is firmly held.`

   Do not start with: `The FK chain should be correct.`

2. Identify dominant sensory channels.

   Candidate channels: visual, motion, sound, timing, interaction, language, accessibility, haptics, rhythm, environment.

3. Identify perceptual levers.

   Candidate levers: contrast, latency, weight, momentum, ownership, rhythm, cadence, scale, brightness, color, silhouette, audio cue, animation anticipation, impact, recovery, spatial consistency, expectation, novelty, friction, feedback.

4. Identify interactions.

   Levers are coupled. Never evaluate them independently.

   Examples:

   - Brightness changes perceived weight.
   - Latency changes perceived quality.
   - Sound changes perceived impact.
   - Carbonation changes perceived acidity.
   - Contrast changes perceived readability.
   - Motion changes perceived ownership.

5. Run human simulation.

   Ignore implementation. Imagine seeing the artifact for one second.

   Ask:

   - What would a first-time human believe?
   - What stands out?
   - What disappears?
   - What feels wrong?
   - What feels inevitable?

6. Describe perception.

   Prefer experience language:

   - It feels...
   - It appears...
   - The brain interprets...
   - The viewer expects...
   - The object reads as...

7. Only now consult metrics.

   Metrics explain. They do not judge.

   Ask which implementation facts explain the observed perception.

8. Generate corrections.

   Corrections should improve perception, not simply improve numbers.

## Domain Modes

### Pose Lab Mode

Never evaluate a pose by transforms first.

Simulate:

- Who owns the weapon?
- Does the hand own it?
- Does the forearm own it?
- Does the world own it?
- Does the camera own it?
- Does it float?
- Does it orbit?
- Does it slide?
- Does it lag?
- Does it feel attached?
- Does the wrist appear to drive it?
- Can the eye immediately identify the attachment point?

Perceptual vocabulary:

- held,
- floating,
- orbiting,
- lagging,
- sliding,
- pinned,
- world attached,
- camera attached,
- broken wrist,
- impossible anatomy,
- confident grip,
- weak grip,
- natural,
- mechanical,
- disconnected.

### UX Mode

Do not ask whether the UI is correct.

Ask:

- What commands attention?
- What is ignored?
- What competes?
- What feels expensive?
- What feels effortless?
- What creates confidence?
- What creates hesitation?
- What feels inevitable?

### Accessibility Mode

Simulate users with different sensory realities:

- low vision,
- blindness,
- motor impairment,
- cognitive load,
- small screens,
- poor lighting,
- fatigue,
- noise.

Question: does the intended experience survive?

## Review Output

Every Sense Simulation review should produce:

- desired experience,
- observed experience,
- contradictions,
- dominant perception,
- supporting metrics,
- recommended changes,
- confidence.

## Failure Modes

Do not confuse passing tests with successful experience.

Do not confuse correct implementation with convincing perception.

Do not optimize proxies. Optimize experience.

## Acceptance Criteria

A successful Sense Simulation changes implementation priorities.

It explains why people perceive what they perceive.

It predicts what humans will notice.

It distinguishes Physical Truth, Visual Truth, Perceptual Truth, and Behavioral Truth without collapsing them together.

## Promotion Gate

Promote only after repeated demonstrations across multiple domains.

Candidate domains:

- Pose Lab,
- UX,
- accessibility,
- game feel,
- environment design,
- audio,
- narrative,
- Drink Simulation,
- interaction design.

If the same simulation grammar consistently predicts human perception better than implementation-first reasoning, the capability is ready for canonical promotion.

Otherwise remain in scratch.

Evidence beats elegance.

## Quartermaster / Prospector Synthesis

Preserve this as strong scratch ore, not canon.

The reusable capability is broader than Visual Truth Authority. Visual Truth Authority establishes that visible artifacts outrank telemetry for visual work. Sense Simulation generalizes the ladder: physical facts create sensory evidence, sensory evidence creates perceptual belief, and perceptual belief produces behavior.

The strongest future proof will be incident records where Sense Simulation changes what gets implemented first.
